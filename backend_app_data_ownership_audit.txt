6. Cross-App Relationships

- `accounts.User` -> `app1.Employee`
  - Type: one-to-one
  - Why: links Maker-Checker user account to HR employee identity
  - Clean or confusing:
    - useful for integration
    - but confusing because HR login still uses `app1.Employee` directly

- `accounts.MakerCheckerRoleAssignment` -> `app1.Employee`
  - Type: one-to-one
  - Why: one HR employee gets one Maker-Checker assignment row
  - Clean or confusing:
    - clean for the new merged Maker-Checker architecture

- `accounts.MakerCheckerRoleAssignment` -> `accounts.User`
  - Type: one-to-one
  - Why: one Maker-Checker auth user is tied to one assignment row
  - Clean or confusing:
    - clean inside Maker-Checker

- `finance.Advance` -> `accounts.User`
  - Type: FK
  - Why: advances belong to maker/checker users, not HR employee rows
  - Clean or confusing:
    - clean for Maker-Checker

- `finance.Expense` -> `finance.Advance`
  - Type: FK
  - Why: expenses are created against an advance
  - Clean or confusing:
    - clean

- `finance.Expense` -> `accounts.User`
  - Type: FK to maker/reviewer/approver
  - Why: workflow is role-based within Maker-Checker
  - Clean or confusing:
    - clean inside Maker-Checker

- `finance.Notification` -> `accounts.User`
  - Type: FK
  - Why: notifications belong to Maker-Checker users
  - Clean or confusing:
    - clean

- `attendance.Attendance` -> `app1.Employee`
  - Type: no FK, only integer `user_id`
  - Why: attendance code assumes `user_id = Employee.id`
  - Clean or confusing:
    - confusing
    - not strongly enforced by database

- `leave.Leave` -> `app1.Employee`
  - Type: no FK, only integer `user_id`, plus copied employee fields
  - Why: leave app copies employee info when request is created
  - Clean or confusing:
    - confusing
    - not strongly enforced by database
    - duplicated employee details inside leave rows

- `login.Profile` -> `accounts.User`
  - Type: FK
  - Why: old profile model built on `AUTH_USER_MODEL`
  - Clean or confusing:
    - confusing because active HR login does not use it

7. Duplicate or Overlapping Data

- Overlap: user/auth identity
  - App 1: `app1.Employee`
  - App 2: `accounts.User`
  - Overlapping data:
    - name
    - email
    - role
    - password/auth behavior
  - Real source of truth:
    - HR identity appears mainly in `app1.Employee`
    - Maker-Checker auth identity appears mainly in `accounts.User`
  - Duplicated/derived/legacy:
    - `accounts.User` is partly derived from `Employee` for HR-linked Maker-Checker users
  - Comment:
    - this is the biggest ownership overlap in the backend

- Overlap: Maker-Checker role
  - App 1: `accounts.User.role`
  - App 2: `accounts.MakerCheckerRoleAssignment.role`
  - Overlapping data:

10. Final Source-of-Truth Summary

The clearest source-of-truth owners in this backend are:
- HR employee master records: `app1.Employee`
- HR candidate pipeline: `app1.Candidate`
- attendance rows: `attendance.Attendance`
- leave request rows: `leave.Leave`
- Maker-Checker auth user rows: `accounts.User`

- petty cash / expense workflow: `finance` models
- shared enums and helper rules: `common`

`app1.Employee`
